## 2. 项目背景

本项目来源于机器人自动化抓取场景中“非结构化、多物体堆叠目标抓取”的实际需求。传统机械臂抓取通常依赖人工示教、固定工装、规则摆放和预设抓取点，适用于单一物体、固定位置、姿态稳定的场景。但在实际生产、仓储分拣、实验室自动化和柔性制造中，目标物体往往存在随机摆放、相互遮挡、堆叠挤压、姿态不确定等情况，传统规则方法难以稳定确定抓取动作和抓取位姿。

结合当前项目代码进度，本项目已经从单纯的 ROS2 仿真方案调整为“真实 PiperX 机械臂优先打通，视觉与 GraspNet 逐步接入”的开发路线。当前工程运行环境为 Windows + Conda `piperx`，机械臂通信链路采用 `python-can`、`python-can-agx-cando`、`pyAgxArm` 和 AGX CANDO CAN 设备，已完成 CAN 设备识别、CAN 帧读取、机械臂连接、状态读取、六轴使能和小幅运动验证。项目工程目录为 `C:\piperx\piperx_graspnet_project`，已经形成配置、机器人控制、坐标转换、GraspNet 输出解析、抓取计划生成、执行脚本和基础测试的代码结构。

在视觉感知层面，项目最终仍以 GraspNet 或同类基于 RGB-D 图像/点云的抓取位姿估计算法为核心，使系统能够通过真实 RGB-D 相机获取场景信息，生成多个六自由度抓取候选，并结合夹爪宽度、抓取评分、工作空间、安全边界和机械臂执行约束，选择可执行的最佳抓取方案。当前代码已经支持从 GraspNet `.npz` 输出中加载 `poses`、`scores`、`widths`，并完成候选筛选、相机坐标系到机械臂基坐标系的矩阵转换、`pregrasp -> grasp -> lift` 抓取计划生成以及真实机械臂执行脚手架。

本项目的典型业务场景包括：仓储物流中的杂乱物品分拣，工业产线中的无序上料，机械零件的料框抓取，实验室自动化中的样本或耗材抓取，以及服务机器人在复杂环境中的目标拾取。这类场景的共同特点是目标位置不固定、物体姿态不确定、多个物体之间存在遮挡和堆叠，系统不能依赖人工预先设定每一个抓取点，而需要根据实时视觉数据自主判断“抓哪里、怎么抓、是否能抓、抓取后如何避障”。

当前痛点主要体现在五个方面。第一，传统视觉方法通常只能识别物体的位置或轮廓，难以直接输出稳定可靠的六自由度抓取位姿。第二，多物体堆叠情况下存在遮挡、接触和重叠，单纯依靠二维图像难以判断物体的空间关系和可抓取性。第三，抓取动作不仅需要视觉识别，还必须考虑机械臂运动范围、夹爪开合约束、碰撞风险和抓取成功率。第四，真实机械臂直接试错成本较高，容易造成碰撞、夹爪损坏或目标物体掉落，因此必须保留 dry-run、低速、小步长和人工确认机制。第五，GraspNet 输出位姿必须经过手眼标定、坐标系约定、TCP 偏移和工作空间检查后，才能转换为真实机械臂可执行动作。

因此，本项目需要视觉或 AI 系统重点解决以下问题：一是从真实 RGB-D 相机采集的图像或点云中感知场景结构，识别目标物体的位置、深度和空间姿态；二是在多物体堆叠情况下生成多个候选抓取位姿，包括抓取点、抓取方向、夹爪姿态和开合宽度；三是对候选抓取进行评分和筛选，判断其可达性、稳定性和碰撞风险；四是将相机坐标系中的抓取位姿通过 `T_base_camera` 转换到 PiperX 机械臂基坐标系；五是通过 `pyAgxArm` 控制真实机械臂和夹爪完成接近、抓取、抬升、放置和复位动作；六是在后续阶段根据需要引入 ROS2 / MoveIt 作为仿真验证、IK 求解、碰撞检测和轨迹规划增强模块。

项目最终目标是建立一套面向非结构化堆叠场景的 PiperX 机械臂智能抓取系统，使机械臂能够不依赖固定工装和人工示教，而是通过摄像头实时获取环境信息，利用 GraspNet 估计抓取位姿，并通过当前已经打通的 AGX CANDO + `pyAgxArm` 控制链路执行抓取动作。系统后续可扩展 ROS2 / MoveIt 仿真和规划能力，以提升复杂场景下的安全性、可验证性和部署适应性。

## 5. 技术方案

### 5.1 总体处理流程

```text
任务触发
  -> 检查 AGX CANDO CAN 设备与 PiperX 通信状态
  -> 读取机械臂状态、关节角、末端位姿和错误状态
  -> 连接 RGB-D 相机并采集 RGB-D 图像/点云
  -> 图像与点云预处理
  -> GraspNet 或抓取检测模型生成候选抓取位姿
  -> 读取或生成 GraspNet 输出 poses/scores/widths
  -> 候选抓取评分、夹爪宽度过滤和排序
  -> 使用 T_base_camera 将相机坐标系位姿转换到 PiperX 基坐标系
  -> 生成预抓取点、抓取点、抬升点和放置点
  -> 执行 dry-run，人工确认位姿和安全边界
  -> 机械臂使能、设置低速、执行接近和抓取动作
  -> 夹爪执行张开、闭合、释放动作
  -> 输出抓取结果、机械臂状态、日志和评估指标
```

### 5.2 系统模块划分

### 5.3 技术路线

第一阶段完成真实机械臂通信和基础控制闭环：在 Windows + Conda `piperx` 环境下安装并验证 `python-can`、`python-can-agx-cando` 和 `pyAgxArm`；通过 AGX CANDO 连接 PiperX；完成 CAN 检测、状态读取、六轴使能、小幅关节运动、小幅末端运动和 dry-run 安全机制。该阶段当前已经基本完成。

第二阶段完成项目工程化和 GraspNet 输出接入：建立 `configs`、`scripts`、`src/piperx_grasp`、`tests` 等工程结构；统一 GraspNet 输出文件格式，支持 `.npz` 中的 `poses`、`scores`、`widths`；实现候选筛选、矩阵转换、抓取计划生成和基础执行脚手架。该阶段当前已完成基础版。

第三阶段完成真实相机、点云和手眼标定：接入 RGB-D 相机，实现 `RgbdCamera.capture()`；保存 RGB、Depth、点云和相机内参；完成相机到 PiperX 基坐标系的手眼标定，将真实 `T_base_camera` 写入 `configs/calibration.yaml`；确认 TCP 偏移 `T_flange_tcp`。该阶段是下一步重点。

第四阶段完成 ROS2 / MoveIt 动作仿真和路径规划：在没有相机的情况下，通过 `scripts/07_prepare_moveit_plan.py --base-pose X Y Z ROLL PITCH YAW` 生成 MoveIt 规划请求 JSON；在 `ros2_ws/src/piperx_moveit_bridge` 中实现 JSON 到 MoveIt 目标位姿的转换、IK 求解、轨迹规划和规划结果输出；在 RViz 或仿真环境中验证 `pregrasp -> grasp -> lift -> place` 动作序列。

第五阶段完成 GraspNet 真实推理和单次抓取闭环：实现或对接 `GraspNetRunner.infer()`，从真实点云生成候选抓取；通过 `scripts/05_run_graspnet_once.py` 检查最佳候选；通过 `scripts/06_pick_once.py` 先 dry-run 再低速执行单次抓取，完成接近、夹取、抬升、放置或复位。随后在多物体堆叠场景中加入更多候选重试、碰撞检测、工作空间约束、失败重感知和日志评估。

### 5.4 异常处理策略

| 异常类型 | 处理策略 |
|---|---|
| CAN 设备未识别或无 CAN 帧 | 检查 AGX CANDO 设备、驱动、通道、bitrate 和 `agx_cando` 后端；通信未恢复前禁止执行机械臂动作。 |
| 机械臂未连接或状态异常 | 停止任务，重新执行状态检查；若 `arm_status`、通信状态或错误状态异常，不进入执行阶段。 |
| 机械臂使能失败 | 循环使能并设置超时；超时后退出任务并提示检查机械臂电源、急停、通信和控制模式。 |
| 当前末端位姿无效 | 等待有效位姿；若超时仍为无效值或 Z 值异常，不生成目标位姿。 |
| 相机数据为空或深度无效 | 重新采集数据；若连续失败，终止任务并记录异常。 |
| 点云数量不足 | 放宽 ROI 或重新采集；仍不足时返回“感知失败”。 |
| 模型未输出有效抓取候选 | 调整抓取阈值或重新采样场景；仍失败时返回“无可抓取目标”。 |
| 坐标转换失败 | 检查 `T_base_camera` 是否存在、矩阵形状是否正确、单位和坐标轴方向是否一致；失败时不执行机械臂动作。 |
| 目标位姿超出安全边界 | 拒绝执行当前候选，切换下一个候选或返回“目标越界”。 |
| MoveIt 或规划器规划失败 | 后续接入规划器后切换下一个候选抓取位姿；候选全部失败时返回“规划失败”。 |
| 碰撞检测失败 | 剔除当前候选抓取位姿，选择下一个候选。 |
| 夹爪闭合失败 | 停止当前抓取流程，执行安全抬升或复位，并记录失败原因。 |
| 抓取后物体滑落 | 重新感知场景，进入重试逻辑。 |
| 脚本或节点异常退出 | 停止任务，保存已有日志，提示需要重新检查设备连接和运行环境。 |

## 6. 算法或规则说明

| 模块 | 处理逻辑 | 关键参数 | 备注 |
|---|---|---|---|
| 图像采集 | 通过真实 RGB-D 相机采集 RGB 图像、深度图和点云，保证图像时间戳和坐标系一致。 | 分辨率、帧率、深度范围、camera_frame | 当前 `RgbdCamera.capture()` 仍待实现。 |
| 点云预处理 | 对点云进行无效点剔除、深度范围过滤、ROI 裁剪、下采样和噪声过滤。 | 深度最小值、深度最大值、ROI 范围、体素大小 | 当前已有 `depth_to_points()` 和 `crop_workspace()` 基础函数。 |
| 抓取检测 | 使用 GraspNet 或同类模型从点云中生成多个 6D 抓取候选，包括抓取中心、姿态、夹爪宽度和评分。 | 候选数量、评分阈值、夹爪宽度范围、模型权重路径 | 当前支持读取 `.npz` 的 `poses/scores/widths`，直接推理未实现。 |
| 候选过滤 | 根据夹爪宽度、抓取高度、姿态角度、工作空间、碰撞风险和机械臂可达性过滤不可执行候选。 | 最小/最大夹爪宽度、安全高度、姿态角限制、工作空间边界 | 当前基础版按分数和最大夹爪宽度过滤。 |
| 候选排序 | 综合模型评分、可达性、碰撞风险、抓取方向和机械臂运动代价进行排序。 | 模型评分权重、路径代价权重、碰撞惩罚权重 | 当前基础版选择 score 最高的候选。 |
| 坐标转换 | 根据手眼标定矩阵，将 camera_frame 下的抓取位姿转换到 PiperX 基坐标系。 | `T_base_camera`、frame_id、时间戳、单位换算 | 当前 `T_base_camera` 是单位矩阵占位，真实抓取前必须替换。 |
| 预抓取点生成 | 在抓取位姿基础上增加安全接近偏移，生成预抓取点。 | `pregrasp_offset_m`、safe_height、抓取方向 | 当前基础版沿 Z 方向增加偏移。 |
| 运动规划 | 当前生成 `pregrasp_pose`、`grasp_pose`、`lift_pose` 并调用 `move_p`、`move_l`；后续可接入 MoveIt 做轨迹规划。 | 速度比例、步长限制、规划时间、工作空间边界 | 真实执行必须先 dry-run 并人工确认。 |
| 夹爪控制 | 抓取前张开夹爪，到达抓取位姿后按候选宽度闭合，抬升后保持夹持，到达放置点后释放。 | open_width、close_width、gripper_width_m、等待时间 | 当前 `GripperClient` 已封装基础接口，需真实夹爪充分验证。 |
| 抓取结果判定 | 根据物体是否随夹爪抬升、目标位姿是否变化、夹爪闭合状态和机械臂状态判断抓取是否成功。 | 抬升高度、物体位移阈值、夹爪闭合阈值 | 真实部署需结合视觉复检或夹爪反馈。 |
| 失败重试 | 抓取失败时重新感知场景或切换候选抓取，达到最大重试次数后返回失败。 | max_retry、候选数量、重新采集开关 | 防止系统无限循环。 |
| 日志追溯 | 保存输入数据、候选位姿、最终位姿、规划结果、执行状态和异常信息。 | 日志路径、保存格式、任务 ID | 当前日志系统尚未完整实现。 |

### 6.1 候选抓取筛选规则

候选抓取位姿需要同时满足以下条件后才能进入真实执行：

1. 抓取评分大于设定阈值。
2. 夹爪开合宽度处于设备允许范围内。
3. 抓取点位于 PiperX 有效工作空间内。
4. 抓取姿态未超出机械臂末端姿态限制。
5. `T_base_camera` 已完成真实标定，且坐标系、单位和旋转方向一致。
6. TCP 偏移或夹爪末端参考点已经确认。
7. 预抓取点、抓取点和抬升点均无明显碰撞风险。
8. 机械臂当前状态正常，六轴已使能，无通信错误和限位错误。
9. dry-run 输出的 `pregrasp_pose`、`grasp_pose`、`lift_pose` 经人工确认安全。
10. 若接入 MoveIt 或其他规划器，则必须能够规划出可执行轨迹。
11. 当前候选优先级高于其他可执行候选，或上一候选执行失败。

### 6.2 坐标转换规则

抓取检测模型输出的位姿默认位于相机坐标系下，执行前必须转换到机械臂基坐标系。当前项目采用的基础转换关系如下：

```text
T_base_grasp = T_base_camera @ T_camera_grasp
```

其中：

```text
T_camera_grasp
  -> GraspNet 输出的 4x4 抓取矩阵

T_base_camera
  -> 手眼标定得到的相机到 PiperX 基坐标系矩阵

T_base_grasp
  -> PiperX 基坐标系下的抓取位姿
```

实际执行前需要将 `T_base_grasp` 转换为 PiperX 控制接口使用的：

```text
[x, y, z, roll, pitch, yaw]
```

实际实现中需要保证 `frame_id`、时间戳、单位、旋转方向、矩阵乘法方向、四元数或 RPY 顺序一致。若外参缺失、矩阵方向不确定、TCP 偏移未确认或坐标系不一致，系统应终止当前抓取任务并输出明确错误信息。后续接入 ROS2 / MoveIt 时，可将链路扩展为：

```text
gripper_pose_camera
  -> camera_depth_optical_frame
  -> camera_link
  -> piperx_base
  -> MoveIt planning frame
```
